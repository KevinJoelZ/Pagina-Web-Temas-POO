Fundamentos de la Programación Orientada a Objetos

En esta página profundizaremos en Encapsulación y Abstracción. Verás definiciones formales, ventajas, buenas prácticas en Java, diagramas ilustrativos y ejemplos prácticos listos para ejecutar.

Encapsulación

Definición: La encapsulación es el mecanismo de ocultar los datos internos de una clase y exponer sólo lo necesario mediante una interfaz pública. Esto protege la integridad del objeto y permite cambiar la implementación interna sin afectar a los consumidores del objeto.

¿Por qué es importante?

  • Protege invariantes del objeto (por ejemplo, que un saldo no sea negativo).
  • Reduce el acoplamiento entre módulos.
  • Permite validar y sanitizar datos en puntos controlados (setters o constructores).
  • Facilita el mantenimiento y la evolución del código.

Modificadores de acceso en Java

Usa private para datos internos; métodos públicos (public) para la interfaz; protected cuando quieras permitir acceso a subclases dentro del paquete; el modificador por defecto (package-private) restringe al paquete. Prefiere mantener el mínimo acceso necesario.

Ejemplo práctico (BankAccount)

public class BankAccount {
    private double balance; // estado encapsulado
    private final String owner;

    public BankAccount(String owner, double initial) {
        this.owner = owner;
        this.balance = Math.max(0, initial);
    }

    public String getOwner() { return owner; }

    public double getBalance() { return balance; }

    public void deposit(double amount) {
        if (amount > 0) balance += amount;
    }

    public boolean withdraw(double amount) {
        if (amount > 0 && amount <= balance) {
            balance -= amount;
            return true;
        }
        return false;
    }

    // método interno no accesible desde fuera
    private boolean hasSufficientFunds(double amount) {
        return amount <= balance;
    }
}

Observa cómo mantenemos la lógica (validaciones y estados) dentro de la clase.

Cards de Refuerzo: Encapsulación

Modificadores de Acceso

private: solo la clase. protected: clase y subclases. public: cualquier lugar. default: mismo paquete. Usa siempre el más restrictivo posible para maximizar la seguridad del código y ocultar detalles de implementación.

Getters y Setters

Proporcionan control sobre atributos privados. Los setters pueden validar datos antes de asignar, garantizando invariantes. Los getters pueden retornar copias defensivas para evitar mutaciones externas no deseadas del estado interno.

Inmutabilidad

Las clases inmutables no pueden cambiar después de su creación. Usa final en atributos y no proporciones setters. Beneficios: seguridad en multi-threading, caché seguro y lógica más predecible.

Cómo usar Encapsulación en la Práctica

  1. Declara atributos como privados: usa private para todos los datos internos que no necesiten acceso externo directo.
  2. Proporciona métodos públicos: crea getters para lectura y setters para escritura, ambos con validación si es necesario.
  3. Valida en los setters: asegúrate de que los datos cumplan con las reglas de negocio antes de ser asignados.
  4. Retorna copias en getters: cuando regreses objetos mutables (como listas), retorna copias para evitar modificaciones externas no deseadas.
  5. Usa constructores para inicializar: inicializa el estado válido del objeto desde el constructor.

Buenas Prácticas de Encapsulación

Principio de Mínimo Privilegio

Solo expone lo necesario. Evita hacer públicos atributos o métodos internos. Cada exposición innecesaria aumenta el acoplamiento y dificulta el mantenimiento futuro del código.

Validación de Datos

Realiza validaciones robustas en constructores y setters. Lanza excepciones con mensajes claros cuando los datos no cumplan con los requisitos. Ejemplo: lanzar IllegalArgumentException si un saldo es negativo.

Documentación Clara

Documenta con Javadoc los métodos públicos indicando qué validan, qué excepciones pueden lanzar y qué retornan. Facilita el uso correcto por otros programadores.

Copias Defensivas

Cuando regreses objetos mutables (listas, arrays), retorna copias en lugar del original. Así evitas que código externo modifique el estado interno sin pasar por los setters validadores.

Invariantes del Objeto

Define condiciones que siempre deben ser ciertas (p. ej., saldo ≥ 0). Garantiza estas condiciones mediante validación en constructores y setters para mantener la integridad del objeto.

Evita Acceso Directo a Datos

Nunca hagas públicos los atributos aunque no tengan setter. Usa getters/setters siempre. Esto permite cambiar la implementación interna sin afectar a los consumidores del código.

Abstracción

Definición: La abstracción reduce la complejidad concentrándose en los aspectos relevantes de un objeto para un contexto determinado. En Java se implementa con interfaces y clases abstractas que especifican contratos y comportamientos sin detallar la implementación.

Abstract class vs Interface (cuando usar cada una)

  • Interface: define un contrato puro; permite múltiples implementaciones; ideal para comportamientos que varias clases no relacionadas deben ofrecer.
  • Clase abstracta: útil cuando existe una relación semántica (is-a) y se desea compartir código común entre implementaciones.

Ejemplo en Java (interfaz y uso)

public interface Pagable {
    double calcularPago();
}

public class Empleado implements Pagable {
    private String nombre;
    private double salario; // salario bruto

    public Empleado(String nombre, double salario) {
        this.nombre = nombre;
        this.salario = Math.max(0, salario);
    }

    public String getNombre() { return nombre; }
    public double getSalario() { return salario; }

    public void setSalario(double salario) {
        if (salario >= 0) this.salario = salario;
    }

    @Override
    public double calcularPago() {
        // ejemplo simple: retención del 10%
        return salario * 0.90;
    }
}

Gracias a la abstracción, cualquier código que dependa de Pagable puede trabajar con empleados, facturas u otros objetos que implementen el contrato sin conocer detalles internos.

Cards de Refuerzo: Abstracción

Clases Abstractas

Define comportamiento común compartido entre subclases relacionadas. Usa abstract para especificar contratos que las subclases deben implementar. Ideal cuando existe relación "is-a" y deseas compartir código funcional.

Interfaces y Contratos

Una interfaz define un contrato puro sin implementación. Permite múltiples implementaciones y desacoplamiento. Ideal para comportamientos que varias clases no relacionadas deben ofrecer de forma polimórfica.

Polimorfismo

Permite objetos de diferentes tipos responder al mismo mensaje de manera apropiada. Gracias a la abstracción, código genérico puede trabajar con muchos tipos distintos sin conocer detalles específicos de cada implementación.

Cómo usar Abstracción en la Práctica

  1. Identifica abstracciones comunes: observa qué comportamientos compartirán varias clases (p. ej., calcular, dibujar, procesar).
  2. Define contratos con interfaces: crea interfaces que especifiquen qué métodos deben implementar las clases.
  3. Usa clases abstractas para comportamiento común: cuando varias clases comparten código e implementación, usa una clase abstracta base.
  4. Programa hacia abstracciones: depende de interfaces o clases abstractas, no de implementaciones concretas, para mayor flexibilidad.
  5. Aplica polimorfismo: escribe código que trabaje con tipos abstractos; en tiempo de ejecución responderán diferente según su tipo real.

Buenas Prácticas de Abstracción

Segregación de Interfaces

Crea interfaces pequeñas y enfocadas en un único propósito. Evita interfaces "gordas" que obliguen a implementaciones innecesarias. Ejemplo: es mejor tener Dibujable y Movible separadas.

Principio Abierto/Cerrado

Diseña clases e interfaces abiertas a extensión pero cerradas a modificación. Nuevas implementaciones deben agregarse sin cambiar código existente. Esto reduce riesgos de romper código funcional.

Principio de Sustitución de Liskov

Las subclases deben poder reemplazar a sus superclases sin romper el funcionamiento. Respeta el contrato de la clase base en tus implementaciones. Evita comportamientos inesperados.

Entiende el Contexto de Uso

Abstrae solo lo relevante para el problema. Una abstracción demasiado genérica es difícil de usar; una demasiado específica reduce reutilización. Balance es clave.

Evita Jerarquías Profundas

Mantén las jerarquías de herencia planas y simples. Jerarquías muy profundas son difíciles de entender y mantener. Si necesitas múltiples comportamientos, considera composición sobre herencia.

Composición sobre Herencia

Cuando sea posible, prefiere composición (tener objetos) sobre herencia. Es más flexible, evita el acoplamiento y permite cambiar comportamiento en tiempo de ejecución.

Ejercicio práctico (resuelto)

Enunciado: crear Empleado, Pagable y una clase Test que muestre los pagos por consola. A continuación la solución completa y comentada.

Solución: código Java

// Pagable.java
public interface Pagable {
    double calcularPago();
}

// Empleado.java
public class Empleado implements Pagable {
    private String nombre;
    private double salario; // salario bruto

    public Empleado(String nombre, double salario) {
        if (nombre == null || nombre.isBlank()) throw new IllegalArgumentException("Nombre inválido");
        this.nombre = nombre;
        this.salario = Math.max(0, salario);
    }

    public String getNombre() { return nombre; }
    public double getSalario() { return salario; }

    public void setSalario(double salario) {
        if (salario < 0) throw new IllegalArgumentException("Salario no puede ser negativo");
        this.salario = salario;
    }

    @Override
    public double calcularPago() {
        // ejemplo simple: descuento de 10% por retenciones
        return salario * 0.90;
    }
}

// Test.java
public class Test {
    public static void main(String[] args) {
        Empleado e1 = new Empleado("Ana", 1200);
        Empleado e2 = new Empleado("Luis", 950);
        Empleado e3 = new Empleado("María", 1500);

        Empleado[] empleados = { e1, e2, e3 };
        for (Empleado e : empleados) {
            System.out.printf("%s -> Salario bruto: %.2f, Pago neto: %.2f\n", e.getNombre(), e.getSalario(), e.calcularPago());
        }
    }
}

Explicación:

La clase valida entrada en el constructor y en setSalario, lo que garantiza el estado válido (encapsulación). La interfaz Pagable abstrae el cálculo del pago; podríamos implementar la interfaz para otros tipos de pagos (por ejemplo, ContratoPorHoras).

Salida esperada (ejemplo):

Ana -> Salario bruto: 1200.00, Pago neto: 1080.00
Luis -> Salario bruto: 950.00, Pago neto: 855.00
María -> Salario bruto: 1500.00, Pago neto: 1350.00

Este ejemplo cubre tanto encapsulación (control de acceso y validaciones) como abstracción (contrato Pagable).
